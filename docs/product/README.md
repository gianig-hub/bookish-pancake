## Purpose

These documents capture **product decisions** — not code. They exist so that:

- Developers understand what they are building and why
- Product decisions are recorded and not lost in chat history
- Disagreements about behaviour can be resolved by reference

## Style

- Keep documents factual and specific
- Use tables and lists over long prose
- Record the "why" as well as the "what"
- Mark things as TODO when they are not yet decided
